Penflow latency clock

Open this page in two windows — one on the host primary display, one on the VDD-extended desktop (which gets streamed to the tablet). Both windows render Date.now() % 1 000 000 in milliseconds, so the digits represent the same OS clock regardless of which process the window is in.

Photograph both screens with a high-speed camera (a phone's 240 fps slow-mo works). The digit difference between the host window and the tablet window is the end-to-end pixel-to-pixel latency in milliseconds.

About the two readings: this clock typically reads ~6–8 ms lower than the in-app HUD's e2e number. Both are correct — the HUD measures from server-side capture timestamp to tablet vsync (so DXGI capture cost is in the budget), while this clock measures host-panel pixel vs tablet-panel pixel and so subtracts the host monitor's own vsync wait. Use the HUD for the conservative number; use this clock for an independent ground truth.

About the Hz reading: this is the browser's requestAnimationFrame tick rate on the host process — driven by Windows DWM, which follows the highest-Hz monitor on your machine, not the one this window is on. It is not a measurement of the tablet panel's effective refresh rate (Movink Pad Pro 14 caps at 165 Hz hardware).
